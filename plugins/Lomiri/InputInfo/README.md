# InputInfo Plugin

Qt 6.8.2 port of the InputInfo plugin from Lomiri. Provides QML access to a unified input device manager, allowing queries and filters for available input devices (mouse, keyboard, etc.) on the system.

**Qt Version**: 6.8.2

**Platform**: Ubuntu 24.04.3 LTS

---

## Overview

Hardware input device monitoring and querying for Lomiri. Provides an API to enumerate input devices (such as mice, keyboards, touchscreens, etc.), filter them by type, and access detailed device properties such as buttons, axis, and specific OS device paths (`/dev/input/...`). Exposes a robust model to QML for dynamically rendering or interacting with connected input devices.

## Build

### Requirements

- Qt 6.8.2
- CMake 3.17+
- Linux with DBus, `libudev`, `libevdev`

### Build Plugin

```bash
cd plugins/Lomiri/InputInfo
rm -rf build
mkdir build && cd build
cmake .. -DCMAKE_PREFIX_PATH=~/Qt/6.8.2/gcc_64
make
```

Output: `libInputInfo.so`, `libInputInfoplugin.so`, `qmldir`

## API

### QDeclarativeInputDeviceModel (QML_ELEMENT)

Main model class for tracking input devices matching an established filter.

**Properties:**

| Property | Type | Access | Description |
|----------|------|--------|-------------|
| `count` | int | Read-only | Number of matched input devices |
| `deviceFilter` | InputType | Read/Write | Filter applied to restrict the listed devices (e.g., Mouse, Keyboard) |

**Methods:**

| Method | Arguments | Returns | Description |
|--------|-----------|---------|-------------|
| `indexOf(path)` | QString path | int | Fetch index of a device by its DBus/OS path |
| `get(index)` | int index | LomiriInputDevice* | Returns the input device at the given model index |

**Signals:**

| Signal | Arguments | Description |
|--------|-----------|-------------|
| `countChanged` | - | Emitted when the total device count changes |
| `deviceFilterChanged` | InputType filter | Emitted when the device filter is updated |

### LomiriInputDevice (QML_ELEMENT)

Class representing a single hardware input device.

**Properties / Methods:**

| Method / Property | Returns | Description |
|-------------------|---------|-------------|
| `name()` | QString | Hardware name of the input device |
| `devicePath()` | QString | OS device path (e.g., `/dev/input/event0`) |
| `type()` | InputTypeFlags | Broad categorization of the device type (Mouse, Keyboard, Touch, etc.) |

## Qt5 to Qt6 Migration

### CMakeLists.txt

**Key Changes:**

- Replaced `add_library(MODULE)` + `add_lomiri_plugin()` with `qt_add_qml_module()`
- Changed `find_package(Qt5 ...)` to `find_package(Qt6 ...)`
- Changed `target_link_libraries()` from `Qt5::` to `Qt6::` targets with `PRIVATE` keyword
- `qt_add_qml_module()` automatically generates `qmldir`

### Header Files

**Pattern (applies to all exposed classes):**
**Qt5 (removed plugin factory code):**
```cpp
// plugin.cpp (DELETED)
void SomePlugin::registerTypes(const char *uri) {
    qmlRegisterType<ClassName>(uri, 1, 0, "ClassName");
    qmlRegisterSingletonType<SingletonClass>(uri, 1, 0, "SingletonClass", ...);
    qmlRegisterUncreatableType<ReadOnlyData>(uri, 1, 0, "ReadOnlyData", "ReadOnlyData cannot be created from QML");
}
```

**Qt6 (macro-based registration):**
```cpp
// Each class header
#include <qqmlintegration.h>

// Regular creatable type
class ClassName: public QObject {
    Q_OBJECT
    QML_ELEMENT     // Can be instantiated from QML
    // ...
};

// Singleton type
class SingletonClass: public QObject {
    Q_OBJECT
    QML_ELEMENT
    QML_SINGLETON   // Only one instance, accessible globally
    // ...
};

// Uncreatable type (accessible but cannot be instantiated from QML)
class ReadOnlyData: public QObject {
    Q_OBJECT
    QML_ELEMENT
    QML_UNCREATABLE("ReadOnlyData cannot be created from QML")
    // ...
};
```


**Why**: Qt6 uses compile-time macro-based registration instead of runtime type registration. This eliminates runtime overhead, reduces binary size, and removes the need for plugin factory classes (`plugin.cpp`).

### Files Deleted

- `plugin.h` — No longer needed (replaced by `QML_ELEMENT` macro)
- `plugin.cpp` — No longer needed (replaced by `QML_ELEMENT` macro)
- `qmldir` — Auto-generated by `qt_add_qml_module()`

## Testing

### QML Testing

Test project located in `Tests/QMLTests/`. Automatically discovers plugin build directory and runs tests.

**Tests validate:**
- `QDeclarativeInputDeviceModel` and `LomiriInputDevice` instantation
- Retrieval of contextual device properties upon completion (`count`, `deviceFilter`, `indexOf()`)
- Asserts that C++ meta type linkages are properly resolved in the QML engine without failure.

### Unit Tests

Test files located in `Tests/UnitTests/`. Contains 2 test files verifying rigorous boundary validation and API bindings.

#### **tst_qinputinfo.cpp**

Tests `QInputDeviceManager` and `LomiriInputDevice` core functionality:

- ✓ Device basics properties defaults (`name`, `devicePath` are initially empty)
- ✓ Device enum flags assignments/bitmasks
- ✓ Device manager filters set/get configurations
- ✓ Signals correctly hooking filter boundaries (`deviceFilterChanged`)
- ✓ Method assertions across `deviceCount()` and maps (`deviceListOfType`, `deviceMap`)
- ✓ C++ Meta properties dynamically queried by `invokeMethod`

#### **tst_qdeclarativeinputdevicemodel.cpp**

Tests QML-exposed data model abstractions:

- ✓ List basics (`rowCount` >= 0) and validation of row bounds
- ✓ Custom Roles are adequately constructed (`NameRole`, `DevicePathRole`) and fetch legitimate bytearray identifiers
- ✓ Signals firing appropriately (such as `countChanged` and filtering) with `QSignalSpy` confirming no duplicated cascades
- ✓ Edge-cases handled cleanly via invokable methods like passing out-of-bounds `get(-1)` or `indexOf("invalid")` defaulting exactly to error returns.
